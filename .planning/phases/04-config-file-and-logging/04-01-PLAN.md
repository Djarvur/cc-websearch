Create `src/lib/config.ts` implementing the following, per CONTEXT.md decisions D-01 through D-07:

**Schema (D-01):** Use `z.strictObject` (same pattern as input.ts) with three optional nested sections:

- `perplexity`: `{ apiKey: z.string().optional(), model: z.string().optional() }` -- entire section `.optional()`
- `retry`: `{ maxRetries: z.number().int().min(0).optional(), baseDelay: z.number().int().min(0).optional(), maxDelay: z.number().int().min(0).optional(), timeout: z.number().int().min(0).optional() }` -- entire section `.optional()`
- `logging`: `{ level: z.enum(['debug','info','warn','error']).optional() }` -- entire section `.optional()`

Export the inferred `Config` type from the schema.

**Hardcoded defaults (per RESEARCH.md):**

- `perplexity.model`: `'sonar'`
- `retry.maxRetries`: `4`
- `retry.baseDelay`: `1000`
- `retry.maxDelay`: `16000`
- `retry.timeout`: `30000`
- `logging.level`: `'info'`

**File reading (D-03, D-05):** Config path is `join(homedir(), '.config', 'websearch', 'config.json')`. Use sync `fs.existsSync` + `fs.readFileSync`. If file does not exist, return `null` silently (no warning, no log). If JSON parse fails, write `[warn] Failed to parse config file\n` to `process.stderr.write()` and return `null`.

**Warning output (D-06, D-07):** Use `ConfigSchema.safeParse()` on the parsed file content. If validation fails, iterate `result.error.issues` and write `[warn] Invalid config at {path.join('.')}: {issue.message}\n` to `process.stderr.write()`. If the raw JSON has keys not in the schema, `z.strictObject` catches them as "Unrecognized key" errors -- this satisfies D-07 automatically.

**CRITICAL: No logger import.** This module writes warnings directly to `process.stderr.write()` to avoid circular dependency with the logger module (RESEARCH.md Pitfall 3).

**Env var override (D-02, D-04):** Define an env var mapping for each config key. The env var names are:

- `WEBSEARCH_PERPLEXITY_API_KEY` -> `perplexity.apiKey`
- `WEBSEARCH_PERPLEXITY_MODEL` -> `perplexity.model`
- `WEBSEARCH_RETRY_MAX_RETRIES` -> `retry.maxRetries`
- `WEBSEARCH_RETRY_BASE_DELAY` -> `retry.baseDelay`
- `WEBSEARCH_RETRY_MAX_DELAY` -> `retry.maxDelay`
- `WEBSEARCH_RETRY_TIMEOUT` -> `retry.timeout`
- `WEBSEARCH_LOGGING_LEVEL` -> `logging.level`

For each config key, resolve independently: if the env var is set and non-empty, use it (coerce strings to numbers for retry fields via `Number()` with NaN validation; validate log level against the enum); else if the file has a value, use it; else use the hardcoded default.

**Export `ResolvedConfig` interface:** All fields fully resolved (no `undefined` except `perplexity.apiKey` which can be `string | undefined`). The retry fields are all `number`, logging.level is the union type, perplexity.model is `string`.

**Export `loadConfig()` function:** Returns `ResolvedConfig`. Calls the file reader, validates with schema, merges env > file > defaults, and returns the resolved object.

## Trust Boundaries

| Boundary                     | Description                                                          |
| ---------------------------- | -------------------------------------------------------------------- |
| config file -> config module | User-controlled file content parsed as JSON and validated            |
| env vars -> config module    | User-controlled environment variable strings coerced to typed values |

## STRIDE Threat Register

| Threat ID | Category               | Component                   | Disposition | Mitigation Plan                                                                                                 |
| --------- | ---------------------- | --------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------- |
| T-04-01   | Information Disclosure | config file (API key)       | mitigate    | Document that config file should have restrictive permissions (0600). Config module does not write, only reads. |
| T-04-02   | Denial of Service      | JSON.parse on config file   | mitigate    | try/catch around JSON.parse; warn and use defaults on failure (D-06)                                            |
| T-04-03   | Tampering              | env var values              | accept      | Env vars are user-controlled by definition. Number coercion validates NaN.                                      |
| T-04-04   | Tampering              | HOME env var -> config path | accept      | Uses os.homedir() which is standard. Fixed path suffix (D-03). Low risk CLI tool.                               |

</threat_model>
