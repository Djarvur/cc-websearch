**Step 1: Refactor src/lib/logger.ts (D-08, D-09, D-10)**

Change the export from a singleton `logger` object to a `createLogger` factory function. The function signature: `createLogger(module: string, level: LogLevel = 'info')`. Remove the module-level `let currentLevel` that reads `process.env.LOG_LEVEL`. Instead, the level is passed as a parameter.

The log format changes from `[level] message\n` to `[${timestamp}] [${level}:${module}] ${message}\n` where `timestamp = new Date().toISOString()`.

Return an object with methods: `debug`, `info`, `warn`, `error` (same as before), plus `setLevel(newLevel: LogLevel)` for runtime level changes.

Do NOT export the old `logger` singleton. The only export is `createLogger`. Also export the `LogLevel` type for consumers.

**Step 2: Update src/lib/duckduckgo.ts and src/lib/fetch.ts (logger import change only)**

These modules only use `logger.debug()`. Change their import from `import { logger } from './logger.js'` to `import { createLogger } from './logger.js'`. Create a module-scoped logger: `const logger = createLogger('ddg', 'info')` and `const logger = createLogger('fetch', 'info')`.

Note: These modules will get their log level from config when wired through the entry points in Plan 03. For now, use the default `'info'` level -- the entry point will set it via `logger.setLevel()` after loading config. Alternatively, accept a logger instance as a parameter. The simpler approach (matching existing patterns): use default level and let the entry point call `setLevel()`. But since these modules create the logger at import time, the cleanest approach is to have the entry point create the logger and pass it, OR to have each module export a function that accepts config.

Simplest change: keep the module-level logger creation with default level. The entry point (Plan 03) will update the level after loading config. This minimizes the refactor surface -- these modules don't read config directly, they just need the right log level.

Practical approach: Have duckduckgo.ts and fetch.ts continue to use a module-scoped logger created with default level. The entry point in Plan 03 will create a properly-leveled logger and either: (a) call setLevel on it, or (b) these modules will be refactored to accept a logger parameter.

For minimum disruption: change the import, create the logger with default level. The level will be set properly when the entry point loads config and updates loggers (Plan 03). This means these modules still have a module-level side effect, but it's the same pattern as before, just with `createLogger` instead of the old singleton.

Keep this interface but have the caller (entry points) always pass the config-derived values as options. This avoids retry.ts needing to import config.ts -- the entry point controls the config flow.
